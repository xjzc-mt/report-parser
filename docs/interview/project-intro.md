## 1. 项目一句话介绍

这是一个纯前端的 ESG 报告处理工具，核心是把 PDF 报告、测试集和 LLM 能力结合起来，形成“提取、验收分析、Prompt 优化、结果导出”的完整闭环。

---

## 2. 30 秒介绍版本

这个项目是一个基于 React + Vite 的浏览器端工具，主要用于从 ESG 报告 PDF 中提取指标，并且我把它从一个单纯的提取工具扩展成了一个完整的测试集工作台。现在它不仅能做 PDF 提取，还能基于测试集对 LLM 结果做验收分析、定位问题、按指标做 Prompt 优化，另外还支持浏览器内 PDF 压缩。项目里我重点做的是测试集工作台、统一分析页以及大数据量分析场景下的性能优化。

---

## 3. 3 分钟介绍版本

这个项目最初是一个前端本地工具，主要功能是上传 PDF 和需求表，然后调用大模型去提取 ESG 指标。后来我把它往“可评估、可优化”的方向扩展了，核心升级是加了测试集工作台。

现在整个项目有 4 个主模块。第一个是提取工作台，负责上传 PDF 和需求表，拆批调用 LLM 做提取。第二个是方法论页，放输入输出规范和系统说明。第三个是浏览器内 PDF 压缩。第四个也是最核心的，就是测试集工作台。

测试集工作台里我又拆成了 3 种模式。完整流程模式是用户自己上传 PDF、测试集和可选的定义文件，先跑提取，再做关联分析，再做 Prompt 优化。快速验收模式适合用户已经有外部 LLM 结果，只需要上传结果和测试集，支持按上传文件真实字段手动配置关联关系，并在分析前做严格校验后直接做验收分析。快速优化模式适合用户已经有一份关联对比文件，想直接对问题指标做 Prompt 优化。

我在这个项目里重点做了统一分析页。它不是简单地把结果列表展示出来，而是先把原始对比结果聚合成测试项级数据，再提供跨报告、按报告、按类型三种树状视角，让用户能一路下钻到某个指标的具体明细。底部还有明细表，支持筛选、长文本展示和导出。

项目里比较有技术含量的一块是性能优化。因为分析结果一大之后，切阈值、切年份、点节点、点筛选都会卡。后来我把分析链路改成了“先建索引，再做子集、再生成树”的方式，同时给明细表做了虚拟滚动和轻量 row model。这样在大数据量下，分析页不会再一点击就整页卡死。

所以如果总结我的核心贡献，我会说是三块：一是把项目从“提取工具”做成“提取 + 验收 + 优化”的闭环；二是设计并实现了统一分析页；三是解决了大结果集下分析页交互卡顿的问题。

## 5. 项目架构怎么讲

如果面试官让你讲架构，你可以按这条线说：

### 5.1 顶层结构

这是一个纯前端应用，没有独立后端服务，主要依赖：

- React 负责 UI 和状态管理
- Vite 做构建
- Mantine 做组件
- IndexedDB 做本地持久化
- `xlsx` 做 Excel 解析和导出
- `pdf-lib` / `pdf.js` 做 PDF 处理
- 外部 LLM API 做提取和优化

### 5.2 代码分层

- `components/`
  - 放页面和 UI 组件
- `services/`
  - 放 PDF 解析、LLM 调用、导出、持久化
- `utils/`
  - 放纯计算逻辑，尤其是分析指标计算和明细模型转换
- `content/`
  - 放静态说明内容

### 5.3 核心链路

完整流程模式的主链路是：

1. 上传 PDF 和测试集
2. 按 `report_name + pdf_numbers` 分组
3. 切页并调用 LLM1 提取
4. 将提取结果和测试集做关联
5. 生成 `comparisonRows`
6. 进入统一分析页做问题定位
7. 选择问题指标，调用 LLM2 做 Prompt 优化
8. 生成 `finalRows` 和优化轨迹并导出

## 6. 这个项目的技术难点怎么讲

## 6.1 难点一：前端里做 PDF 提取和多流程编排

难点在于这个项目不是普通管理后台，它要处理：

- PDF 文件上传
- Excel 解析
- 大模型调用
- 结果导出
- 本地缓存和断点恢复

而这些都发生在浏览器端，所以需要把文件处理、异步进度、失败重试、UI 状态都组织好。

你可以强调：

- 这是一个“前端承担业务编排”的项目
- 不只是写页面，而是要设计完整的数据流和失败处理

## 6.2 难点二：分析页大数据量卡顿

这个是你最值得讲的点。

你可以这样说：

一开始分析页是基于原始 `comparisonRows` 反复全量计算的，用户每次切年份、切阈值、切节点、点筛选，都会触发大量聚合计算和大表渲染，结果就是页面明显卡顿。后来我把它改成了分层数据流：先构建分析索引，再基于年份取子集，再根据阈值生成模型，树节点查找改成映射表，明细表则改成轻量 row model + 虚拟滚动。这样大数据量下交互就顺畅很多。

## 6.3 难点三：长文本和不同类型明细的展示

统一分析页底部的明细表不是简单表格，因为：

- 文字型值可能很长
- 数值型、货币型、强度型字段结构不一样
- 行高不能统一

所以后来做了：

- 文字型单独更高行高
- 其他类型用较紧凑的固定高度
- 文字型值最多显示 12 行，hover 看全量
- 明细表仍然保持虚拟滚动

这个点能体现你对“性能和可读性折中”的设计能力。

## 6.4 难点四：本地持久化与断点恢复

因为没有后端，项目里很多中间结果都要自己在浏览器里持久化，比如：

- 上传的 PDF / Excel
- 缓存的切页
- 阶段结果
- 快速验收的分析结果
- 未完成运行状态

你可以强调这不是简单 localStorage，而是 IndexedDB 分层存储。

- 不是只做提取

## 9. 高频追问与回答

## 9.1 面试官：这个项目为什么要做成纯前端？

你可以答：

一方面这个工具面向的是本地文件处理场景，很多 PDF 和测试集比较敏感，放在浏览器端本地处理更合适；另一方面早期产品目标是快速验证提取和验收流程，所以前端直连大模型 API、前端本地导出 Excel 的方式能更快落地。

## 9.2 面试官：为什么分析页会卡？你是怎么定位的？

你可以答：

本质原因是每次交互都在做两件重活：一是从原始对比结果全量重算分析模型，二是整张明细表重新渲染。我是先从用户反馈的卡顿场景入手，对照组件数据流看哪些状态变化会触发整页重算，然后把问题收敛到分析模型全量重建和大表渲染这两条链路上，最后分别用预索引和虚拟滚动解决。

## 9.3 面试官：为什么不用服务端来做分析聚合？

你可以答：

这个项目当前定位是本地工具，强调敏感文件不出本地，而且很多分析逻辑本身是围绕前端已经拿到的 `comparisonRows` 展开的，所以这一版优先选择在前端做聚合和缓存。后续如果场景变成多人协作或大规模批处理，再考虑把分析层或任务调度层迁到服务端更合理。

## 9.4 面试官：你做虚拟滚动时最大的难点是什么？

你可以答：

不是“把表变成虚拟列表”本身，而是这个表不是每行同高。文字型、数值型、货币型、强度型的字段结构都不一样，如果直接做简单虚拟列表，滚动定位会错。所以我最后采用的是“按类型固定几档高度”，先预计算每行高度和累计 offset，再按窗口渲染。

## 9.5 面试官：这个项目里你最满意的一次改动是什么？

你可以答：

我最满意的是统一分析页这块，因为它同时解决了两个问题：一是把分析能力从零散统计升级成可下钻的结构化视图，二是把原来一点击就卡的分析交互改成可用的状态。这个改动既有业务价值，也有工程价值。

## 10. 如果面试官继续深挖，你可以主动提的点

### 可主动展开 1：测试项级口径

你可以提：

我在分析层不是直接按原始结果行统计，而是先聚合成测试项级对象，因为一个测试项可能对应 0 条、1 条或多条输出。如果直接按行统计，重复摘录、幻觉、通过率的口径会乱。

### 可主动展开 2：类型口径分离

你可以提：

我在分析汇总层把货币型并入数值型，避免树结构过散；但在明细展示层还是保留货币型，因为它比普通数值型多一个货币字段。这是汇总视角和展示视角分离的一个例子。

### 可主动展开 3：为什么没做分析状态持久化

你可以提：

我刻意没把分析页的选中节点、筛选条件、滚动位置全部持久化，因为当时更核心的问题是交互卡顿，不是刷新恢复。先把主线程计算和渲染压下来，比做重度状态恢复更重要。
