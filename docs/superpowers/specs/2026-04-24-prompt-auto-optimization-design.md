## 2. 结论先行

`Prompt 自动优化` 是平台未来最重要的核心能力之一，但不是当前阶段的第一优先级。

合理顺序应为：

1. 先补平台底座
2. 再做 `Prompt 自动优化 V1`
3. 然后扩展到 few-shot 优化
4. 最后进入联合优化

原因很简单：

- 没有统一实验对象模型，优化结果无法沉淀
- 没有统一评估引擎，系统不知道“什么叫变好”
- 没有运行追踪与回放，优化过程不可复现

因此，`Prompt 自动优化` 的正确建设方式不是一上来追求“大一统通用优化器”，而是分阶段演进。

---

## 3. 概念边界

为了避免后续讨论混淆，先固定几个定义。

### 3.1 什么是 Prompt

在本项目里，`Prompt` 优先指：

- `system prompt`
- `user prompt` 模板
- 输出格式约束
- 规则说明文本

这部分可以统称为“提示词文本本身”。

### 3.2 什么是 few-shot

few-shot 是放进上下文的示例样本，虽然运行时会随 Prompt 一起发给模型，但在工程上应该视为独立资产，而不是普通文案。

原因：

- 它依赖数据集质量
- 它需要选择策略
- 它有数量、顺序、格式等独立优化问题

### 3.3 什么是 chunk 策略

chunk 策略不属于 Prompt 文本本身。

它指的是：

- 文档如何切分
- 切多少
- 选哪些段
- 以什么顺序送给模型

它属于文档预处理 / 检索 / 运行编排层，而不是提示词层。

## 4. 当前平台现状判断

当前平台已经具备：

- `Prompt 快速迭代`
- `模型结果验收`
- 现有 `Prompt 自动优化` 页面雏形
- 文档上传、页码控制、结果分析、历史记录等局部能力

但还不具备真正做“平台级自动优化”的前提条件：

- `PromptVersion / Dataset / Run / EvalReport / Artifact` 还未统一对象化
- 评估体系还没有沉成统一 `Eval Engine`
- 运行记录与回放还不完整
- 文档处理策略还没有统一抽象

因此，当前页面里的 `Prompt 自动优化` 更接近“定向辅助优化入口”，而不是通用优化系统。

---

## 5. 总体优先级

`Prompt 自动优化` 在全平台优化事项中的排序如下。

### P0：必须先完成的平台底座

1. 统一实验对象模型
2. 统一评估引擎
3. 统一运行记录与追踪

只有这三项成立后，自动优化才有可靠地基。

### P1：Prompt 自动优化 V1

只优化“提示词文本本身”：

- `system prompt`
- `user prompt` 模板
- 输出约束文案

这是第一版正式优化器的边界。

### P2：few-shot 优化

在 V1 跑稳后，引入示例样本的自动挑选与排序。

### P3：联合优化

最后再进入真正的通用优化：

- 提示词文本
- few-shot
- chunk 策略
- 运行参数

---

## 6. 为什么 V1 只优化提示词文本本身

这是当前阶段最合理的切入点。

### 6.1 原因一：可解释性最好

用户最容易理解：

- 原 Prompt 是什么
- 新 Prompt 改了什么
- 为什么效果提升

### 6.2 原因二：风险最低

只改提示词文本，不改上游数据管道，不会把问题扩大到：

- 文档切分错误
- 检索召回偏差
- few-shot 样本污染

### 6.3 原因三：最适合做平台第一版闭环

第一版最重要的是打通这条链路：

`Baseline Prompt -> 候选 Prompt -> 批量评估 -> 排序 -> 人工确认 -> 版本沉淀`

这条链路先跑通，比一开始上复杂联合搜索更重要。

---

## 7. V1 目标形态

`Prompt 自动优化 V1` 的目标不是“神奇地自动写出最优 Prompt”，而是把平台升级成一个可靠的 Prompt 优化实验系统。

### 7.1 输入

- 一个基线 Prompt
- 一个数据集
- 一个明确评估指标
- 一个指定模型
- 一个优化预算

### 7.2 输出

- 若干候选 Prompt
- 每个候选 Prompt 的评估结果
- 与基线 Prompt 的对比结果
- 推荐最佳版本
- 可回放的优化记录

### 7.3 人机分工

系统负责：

- 候选生成
- 批量评测
- 排序与总结

用户负责：

- 选择 baseline
- 确认评估口径
- 审核最终采用版本

---

## 8. V1 核心流程

### 8.1 选择基线

从平台中选定：

- 一个 Prompt 版本
- 一个数据集
- 一个评估器

### 8.2 生成候选

优化器基于基线 Prompt 生成多份候选改写。

候选生成暂不追求复杂搜索算法，先支持：

- 基于错误类型的定向改写
- 基于规则的结构强化
- 基于 LLM 的候选重写

### 8.3 批量评测

对每个候选 Prompt 跑同一份数据集，并输出统一指标。

### 8.4 排序与归因

平台给出：

- 总体得分排序
- 相对 baseline 的提升幅度
- 典型改善样本
- 典型退化样本

### 8.5 人工确认

最佳候选不能自动替换生产 Prompt，必须人工确认后才沉淀为新版本。

---

## 9. V1 不做什么

为了避免范围失控，第一版明确不做以下内容：

- 不做 chunk 策略搜索
- 不做 few-shot 自动检索与联合搜索
- 不做多模型联合优化
- 不做完全无人值守的生产替换
- 不做黑箱式“只给一个结果，不解释原因”

---

## 10. V2 与 V3 演进方向

### 10.1 V2：few-shot 优化

在 V1 稳定后，扩展到：

- 示例挑选
- 示例顺序
- 示例格式模板
- 示例数量控制

### 10.2 V3：通用联合优化

最终形态才进入：

- Prompt 文本
- few-shot
- chunk 策略
- 运行参数

这里才接近真正意义上的“通用优化方案”。

---

## 11. 依赖关系

`Prompt 自动优化` 启动前，至少应依赖以下平台能力就位：

1. Prompt 版本化
2. 数据集对象化
3. 评估器对象化
4. 运行结果可追踪
5. 优化结果可持久化

如果这些能力未完成，自动优化只能是页面级辅助工具，不能视为平台级核心模块。

---

## 12. 成功标准

专项设计是否成功，以以下标准衡量：

1. 用户能清晰理解优化对象和阶段边界
2. 第一版可以稳定复现“基线 -> 候选 -> 评估 -> 推荐”的闭环
3. 每次优化都有可回放记录
4. 最终采用版本必须可追溯
5. 后续可以平滑扩展到 few-shot 与联合优化，而不是推倒重来

---

## 13. 最终决策

关于 `Prompt 自动优化`，当前确认的方案是：

1. 它是平台未来最重要的能力之一，但不是第一批落地功能
2. 先做平台底座，再做优化器
3. 优化器第一版只优化提示词文本本身
4. few-shot 优化与 chunk 联合优化放在后续阶段
5. 后续所有关于这块的需求、变更和争议，以本文件为主文档持续迭代

---

## 14. 实现状态

截至当前版本，已经落地的内容是：

- 本地 `PromptAsset / PromptVersion / Dataset / Run / Trace` 对象模型
- Prompt 自动优化运行记录与历史回放的本地持久化
- `Prompt 自动优化` 页面升级为 `优化配置 / 历史运行 / 结果审核` 三段式结构
- 候选 Prompt 需要人工确认后，才能应用为新版本

当前仍未落地的内容是：

- few-shot 自动优化
- chunk 策略联合优化
- 后端控制面
- 平台级统一 `Eval Engine`

也就是说，当前已经进入 `P0 + P1` 的前半段：  
先把本地实验对象、历史记录、人工确认闭环打通，再继续往更通用的自动优化器推进。
