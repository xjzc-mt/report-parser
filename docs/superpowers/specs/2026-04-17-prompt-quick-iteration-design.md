# Prompt快速迭代页面改版设计

## 1. 背景

`Prompt快速迭代` 当前仍沿用“完整流程模式”的结构，依赖测试集文件、定义文件、阶段一提取和阶段二优化。这和页面的新目标不一致。

这一页的新定位不是完整流程编排，而是：

> 快速修改 Prompt，并在多份 PDF 上立即验证提取效果，比较同一套 Prompt 在不同文件中的表现。

页面需要服务的核心动作是：

1. 改 Prompt。
2. 选多份 PDF。
3. 按文件精确指定页码或直接跑全文。
4. 看每份文件的结构化结果和模型原始返回。
5. 保留历史结果，便于回看上一版 Prompt。

## 2. 改版目标

本次改版目标如下：

1. 将 `Prompt快速迭代` 从“完整流程模式”改为轻量 Prompt 实验页。
2. 支持一次上传多份 PDF，并对每份 PDF 单独指定页码。
3. 支持输入一个统一的提取目标名称。
4. 支持分别输入系统提示词和用户提示词。
5. 页面只负责展示模型原始回复，并在能解析出合法 JSON 时自动展示结构化结果。
6. 结果区要天然适合横向比较同一套 Prompt 在不同文件中的表现。
7. 保留历史运行记录，用于 Prompt 优化前后人工对比。

## 3. 非目标

本次不做以下内容：

1. 不支持一次配置多个提取目标。
2. 不做测试集评分、字段级对比或命中率统计。
3. 不做历史版本自动 diff。
4. 不强制固定输出 schema。
5. 不做 Prompt 自动优化算法。
6. 不和 `模型结果验收`、`Prompt自动优化` 形成自动联动。

## 4. 页面定位

`Prompt快速迭代` 应被重新定义为：

> 一个围绕单个提取目标、面向多 PDF 快速试跑和结果比对的 Prompt 实验台。

它不是“完整流程模式”的轻量入口，而是独立的实验页。

它的使用场景是：

- 用户想验证某条 Prompt 是否能稳定提取目标内容。
- 用户想观察 Prompt 在不同 PDF 上是否同样生效。
- 用户想缩小上下文，只对指定页跑模型。
- 用户想回看上一次 Prompt 版本的返回结果。

## 5. 页面结构

页面采用单页四段式结构。

### 5.1 实验配置区

展示并编辑本次运行的核心配置：

- `名称`
- `系统提示词`
- `用户提示词`
- 当前模型/Provider 摘要

说明：

- `名称` 表示这次实验验证的统一提取目标。
- 页面不根据 `名称` 自动生成 schema，只把它作为上下文信息。

### 5.2 文件与页码区

支持上传多份 PDF，并为每份 PDF 单独配置页码。

每个文件一行，至少展示：

- 文件名
- 文件大小
- 页码输入框
- 删除文件操作

页码规则：

- 支持单页：`12`
- 支持多页：`12,15,18`
- 支持范围：`12-15`
- 支持混合：`12-15,18,20`
- 留空表示全文

### 5.3 结果区

结果区不做大卡片切换，固定分成上下两层。

上半区为 `结果汇总`：

- 每个 PDF 一行
- 直接展示横向对比信息

下半区为 `模型原始返回`：

- 按 PDF 顺序展示原始回复全文
- 若成功解析出 JSON，则同时展示完整 JSON

### 5.4 历史区

结果区最外层使用 Tab：

- `当前结果`
- `历史记录`

`历史记录` 用于回看之前某次 Prompt 试跑的完整快照，不做复杂 diff，只保证可读、可比较。

## 6. 输入与执行规则

### 6.1 输入字段

本次页面仅保留以下输入：

- 多 PDF 上传
- 单个统一名称
- 系统提示词
- 用户提示词
- 每个 PDF 独立页码

以下旧输入不再保留：

- 测试集文件
- 指标定义文件
- 阶段一/阶段二控制项
- 完整流程相关配置

### 6.2 执行方式

点击运行后，以 PDF 为粒度逐个执行：

1. 为每个 PDF 生成单独任务。
2. 若填写页码：
   - 先抽取指定页生成子 PDF。
   - 再将子 PDF 发给模型。
3. 若未填写页码：
   - 直接发送整份 PDF。
4. 每个 PDF 单独保存运行结果。
5. 页面聚合所有 PDF 的结果形成汇总。

设计理由：

- 这页的目的不是高吞吐批处理，而是快速、清楚地看同一套 Prompt 在不同 PDF 上的表现。

## 7. 结果展示设计

### 7.1 汇总表

汇总表是结果区的默认重点。

每个 PDF 一行，建议展示：

- 文件名
- 配置页码
- 实际发送范围（全文 / 指定页）
- 运行状态
- 是否解析出 JSON
- 解析结果摘要
- token 消耗
- 耗时

其中“解析结果摘要”不要求固定字段，只根据解析结果做简短摘要：

- 若是对象，显示顶层 key
- 若是数组，显示长度和首项概览
- 若无合法 JSON，显示“未解析出 JSON”

### 7.2 原始返回区

原始返回区按 PDF 依次展示：

- 文件名
- 实际发送范围说明
- 原始回复全文
- JSON 解析结果（若存在）

页面不隐藏原始返回，也不把 JSON 解析结果视为唯一正确结果。

这样用户可以同时判断：

- Prompt 是否让模型答对方向
- 模型是否按要求输出了结构化 JSON
- 多份 PDF 的表现是否一致

## 8. JSON 解析策略

页面不定义固定 schema，但会尝试自动解析模型回复中的 JSON。

解析顺序如下：

1. 先尝试把整段回复直接作为 JSON 解析。
2. 若失败，再尝试提取 Markdown 代码块中的 JSON。
3. 若仍失败，再尝试提取首个看似完整的 `{...}` 或 `[...]` 片段。

解析结果分三种状态：

- `success`：成功解析出合法 JSON
- `not_found`：回复中没有可解析 JSON
- `invalid`：发现疑似 JSON 但无法解析

注意：

- `not_found` 和 `invalid` 不等于运行失败。
- 运行失败只用于模型调用失败、页码错误、抽页失败等场景。

## 9. 历史记录设计

历史记录保存的是一次完整实验快照，而不是只保存最终文本。

每条历史记录至少包含：

- `id`
- `createdAt`
- `name`
- `systemPrompt`
- `userPrompt`
- `modelName`
- `providerType`
- `results`
  - `fileName`
  - `pageSpec`
  - `resolvedPages`
  - `scopeLabel`
  - `status`
  - `rawResponse`
  - `parsedJson`
  - `jsonParseStatus`
  - `usage`
  - `durationMs`

历史页展示方式：

- 左侧：历史运行列表
- 右侧：选中记录的汇总 + 原始返回

第一版不做“把两次历史并排 diff”的复杂交互，但数据结构应支持未来扩展。

## 10. 持久化范围

本页的持久化分为两层。

### 10.1 草稿配置

用于用户下次回来继续编辑。

建议持久化：

- 名称
- 系统提示词
- 用户提示词
- 已上传 PDF 文件
- 每个 PDF 的页码配置

### 10.2 历史记录

用于保存每次运行快照。

建议：

- 保存最近若干次，例如 20 次
- 新运行追加写入
- 草稿配置和历史记录分开存储

恢复规则：

- 页面加载时先恢复草稿配置
- 历史记录作为独立数据源展示
- 新运行不会覆盖草稿

## 11. 错误处理

### 11.1 输入校验

开始运行前需要拦截：

- 未上传 PDF
- 名称为空
- 系统提示词和用户提示词同时为空
- 页码格式非法
- 指定页码超出 PDF 范围

### 11.2 运行错误

运行过程中允许单文件失败，不阻断整批结果。

单文件可能出现的错误：

- 抽页失败
- 模型调用失败
- 模型返回为空

处理方式：

- 汇总表中按文件显示状态
- 顶部只显示简短整体错误提示
- 原始返回区对失败文件展示错误原因

## 12. 组件与服务边界

本次建议拆成三层。

### 12.1 页面层

负责：

- 输入区
- 文件列表与页码交互
- 当前结果展示
- 历史记录展示

### 12.2 服务层

新增轻量 Prompt 实验服务，负责：

- 组装任务
- 校验页码
- 抽取指定页
- 调用 LLM
- 解析 JSON
- 汇总每个 PDF 的结果

### 12.3 持久化层

新增草稿与历史记录存取接口，负责：

- 保存和恢复草稿配置
- 写入和读取历史记录
- 控制历史数量上限

## 13. 现有能力复用

本次实现应尽量复用现有能力：

- 复用已有 PDF 抽页逻辑
- 复用已有 LLM 调用与 Provider 适配逻辑
- 复用已有文件持久化能力

但需要避免继续复用“完整流程模式”的业务编排。

原因：

- 旧编排围绕测试集、定义文件和完整流程阶段构建
- 新页面只需要轻量实验能力
- 如果继续挂在旧编排上，会让页面边界继续混乱

## 14. 验证要求

本次改版完成后至少需要验证：

1. 单 PDF + 全文运行成功。
2. 单 PDF + 指定页码运行成功。
3. 多 PDF + 不同页码配置运行成功。
4. 某个 PDF 页码非法时，页面能正确阻断或标记错误。
5. 模型返回合法 JSON 时，页面能正确解析并展示。
6. 模型返回普通文本时，页面仍能展示原始回复。
7. 历史记录能成功写入并回看。
8. 草稿配置刷新后可以恢复。
9. `npm test` 通过。
10. `npm run build` 通过。

## 15. 完成标准

本次改版完成的标准是：

- `Prompt快速迭代` 已从完整流程模式独立出来
- 页面围绕“快速试 Prompt + 多文件比较”展开
- 用户可以对每份 PDF 单独指定页码
- 页面能同时展示汇总结果和模型原始返回
- 页面能自动解析并展示合法 JSON
- 页面保留历史实验记录，支持回看

如果以上条件都满足，就说明这一页已经从“旧完整流程入口”变成了真正的 Prompt 实验台。
