# LLM Lab 第一阶段改版设计

## 1. 背景

当前项目已经不只是一个“ESG 报告摘录工具”，而是逐渐形成了围绕 LLM 提取、验收、分析、优化和预处理的实验闭环。但现有信息架构仍然强绑定旧定位：

- 品牌名仍然是 `ESG报告摘录大师`
- 顶层导航直接暴露旧工具入口
- 多个能力虽然已经存在，但没有被组织成通用实验平台的结构
- ESG 场景仍然像产品本体，而不是一个已跑通的样例

这会阻碍项目继续演进为更通用的 LLM 工程化实验平台。

## 2. 第一阶段目标

第一阶段不是重写产品能力，而是安全完成平台壳层重组，让项目从“垂直工具”切到“通用实验平台”语义。

阶段目标：

1. 将产品品牌升级为 `LLM Lab`。
2. 将项目定位明确为“大语言模型工程实验室”。
3. 将现有能力重组为三个工作台和一个独立文档入口。
4. 保持现有成熟能力可用，只调整入口和挂载关系。
5. 为后续通用化和深改页面预留稳定结构。

## 3. 平台定位

### 3.1 最终方向

本项目未来目标不是“业务交付平台”，而是优先演进为：

**通用 LLM 实验平台**

核心关注点是：

- 预处理策略实验
- Prompt 快速迭代
- 模型结果验收
- Prompt 自动优化
- 实验过程的工程化组织与复用

### 3.2 第一阶段的产品定义

第一阶段应把当前项目重新定义为：

> 一个以 `LLM Lab` 为品牌、当前内置 ESG 实验样例的大语言模型工程实验平台。

这里的关键转变是：

- ESG 不再是产品唯一主题
- ESG 是当前最成熟、最完整的一个实验样例
- 平台结构需要支持未来继续接入新的任务类型与实验页

## 4. 设计原则

第一阶段必须遵守以下原则：

1. **平台壳先行**
   先改品牌、导航、工作台结构和页面挂载，再考虑深改内部能力。
2. **业务状态不搬家**
   原有业务状态尽量继续留在原页面中，避免阶段一顺手演变成全面架构重写。
3. **旧能力新语义**
   现有页面能力先以新的工作台语义对外呈现，不急于第一阶段消除所有旧命名和旧结构痕迹。
4. **高风险页隔离**
   `Prompt快速迭代` 对应的完整流程页后续会大改，因此第一阶段必须单独隔离它的入口和边界。
5. **可扩展优先**
   新结构要天然支持以后继续接入新的实验页，而不是继续围绕 ESG 单场景生长。

## 5. 信息架构

### 5.1 品牌区

品牌区替换为：

- 主标题：`LLM Lab`
- 副标题：`大语言模型工程实验室`

说明：

- `LLM Lab` 只是品牌名，不是一级导航入口。
- 副标题承担定位说明，帮助用户理解该项目不再是 ESG 单点工具。

### 5.2 一级导航

顶层导航保持和当前相似的结构形式，继续使用顶部一级标签，但导航内容改为：

- `测试集工作台`
- `线上验证工作台`
- `数据预处理工作台`
- `说明文档`

说明：

- 不再将 `PDF压缩`、`摘录工作台` 等旧具体功能直接暴露为顶层入口。
- `说明文档` 取代原 `方法论 / Methodology` 入口名称，统一用中文表达。

### 5.3 工作台结构

#### 测试集工作台

采用页面内二级子标签：

- `Prompt快速迭代 · 原完整流程模式`
- `模型结果验收 · 原快速验收模式`
- `Prompt自动优化 · 原快速优化模式`

它代表未来平台中的实验主闭环区，是后续最值得持续扩展和通用化的工作台。

#### 线上验证工作台

不设子页面，直接承载现有摘录工作台能力。

页面语义为：

- 新名称：`线上验证工作台`
- 辅助标注：`原摘录工作台`

#### 数据预处理工作台

采用页面内二级子标签：

- `Chunking测试`
- `PDF压缩 · 原 PDF压缩`
- `Token统计`

其中：

- `Chunking测试` 第一阶段接入真实页面能力
- `PDF压缩` 直接复用现有实现
- `Token统计` 第一阶段只做清晰占位，不实现完整能力

#### 说明文档

保留独立一级入口，不并入某个工作台。

它继续承载：

- 平台说明
- 输入输出规范
- 使用边界
- 背景方法说明

## 6. 页面映射关系

第一阶段的页面映射关系固定为：

### 测试集工作台

- `Prompt快速迭代` = 当前完整流程模式
- `模型结果验收` = 当前快速验收模式
- `Prompt自动优化` = 当前快速优化模式

### 线上验证工作台

- `线上验证工作台` = 当前摘录工作台

### 数据预处理工作台

- `Chunking测试` = 外部接入的 PDF 切割器页面
- `PDF压缩` = 当前 PDF 压缩页
- `Token统计` = 第一阶段占位页

### 说明文档

- `说明文档` = 当前 Methodology 页面能力，以新入口名继续提供

## 7. 组件边界设计

### 7.1 顶层应用壳

`App.jsx` 第一阶段应只负责：

- 一级导航状态
- 根据一级导航挂载对应工作台或说明文档页

不再直接承载具体业务流程细节。

### 7.2 品牌与顶层导航

`Header.jsx` 负责：

- 渲染品牌区 `LLM Lab`
- 渲染副标题 `大语言模型工程实验室`
- 渲染一级导航

它不再负责旧工具语义的展示。

### 7.3 工作台容器层

新增三个容器组件：

- `TestSetWorkbench.jsx`
- `OnlineValidationWorkbench.jsx`
- `DataPreprocessingWorkbench.jsx`

这些容器只负责：

- 页面标题与说明
- 二级子标签切换
- 挂载已有能力页

它们不直接实现复杂业务逻辑。

### 7.4 能力适配层

为避免旧实现和新平台语义直接耦合，第一阶段建议引入适配页：

- `PromptQuickIterationPage.jsx`
- `ModelValidationPage.jsx`
- `PromptAutoOptimizationPage.jsx`
- `OnlineValidationPage.jsx`
- `PdfCompressionPage.jsx`
- `ChunkingTestPage.jsx`
- `TokenEstimationPage.jsx`

这些适配页的职责是：

- 接收新平台的页面语义
- 继续挂载旧实现或外部实现
- 为第二阶段的进一步重构提供稳定替换点

### 7.5 关键隔离要求

`Prompt快速迭代` 对应的完整流程能力后续会大改，因此第一阶段必须满足：

- 它有独立页面入口
- 它不再继续依赖旧的模式切换语义对外暴露
- 它与 `模型结果验收 / Prompt自动优化` 在组件边界上隔离

这样后续替换时，只需要重写 `PromptQuickIterationPage` 对应内部，而不会牵连整个测试集工作台结构。

## 8. 状态与数据流设计

第一阶段的数据流策略是：

**导航状态上移，业务状态留在原页面。**

### 8.1 `App.jsx`

只管理一级导航状态，例如：

- `test-workbench`
- `online-validation`
- `data-prep`
- `docs`

### 8.2 `TestSetWorkbench`

只管理：

- 当前二级子页面
- 子页面之间的切换
- 必要的跨子页跳转桥接

例如当前已有的链路：

- `模型结果验收` 分析后跳转到 `Prompt自动优化`，并带预选优化范围

这类最小桥接状态可以上移到工作台容器。

### 8.3 `OnlineValidationWorkbench`

不设子页面，因此只需要极少量壳层状态，主要是语义包装，不额外接管业务状态。

### 8.4 `DataPreprocessingWorkbench`

管理：

- `chunking`
- `pdf-compress`
- `token-estimation`

三个子页面的切换状态。

### 8.5 原页面业务状态不迁移

第一阶段不应上移以下业务状态：

- `ExtractorTab` 的上传文件、设置、进度、结果
- `QuickValidationMode` 的验收文件、手动关联字段配置、分析结果
- `QuickOptimizationMode` 的优化输入、范围、导出结果
- `CompressorTab` 的压缩输入和下载结果
- 完整流程页当前内部运行状态

原因是：

- 第一阶段的重点是壳层重组，不是业务重写
- 保持业务状态在原页内部可以显著降低回归风险

## 9. `Chunking测试` 接入策略

`Chunking测试` 来自外部目录：

`/Users/michael_drj/Desktop/report-parser`

第一阶段的接入原则是：

1. 以独立页面形式接入
2. 不直接把外部逻辑揉进当前项目已有 service 体系
3. 先建立页面适配层，再逐步决定后续是否合并底层逻辑

推荐方式：

- 新建 `ChunkingTestPage.jsx`
- 外层采用当前项目的页面布局和视觉风格
- 内部封装迁移后的 PDF 切割器能力

这样能避免两个来源的逻辑在第一阶段被粗暴混合。

## 10. `Token统计` 第一阶段定义

`Token统计` 是明确在路线中的新能力，但第一阶段只做占位。

占位页应明确说明：

- 它未来用于估计输入内容的 token 量
- 目标支持多种文件类型和纯文本输入
- 本阶段暂不实现正式统计能力

第一阶段不做：

- 完整 token 估算算法
- 各文件类型解析适配
- 模型差异化 token 口径对比

## 11. 实施顺序

第一阶段按以下顺序推进最安全：

1. 改品牌与一级导航
2. 新增三个工作台容器
3. 先完成测试集工作台三子页挂载
4. 平移线上验证工作台
5. 落位数据预处理工作台
6. 接入 `Chunking测试`
7. 新增 `Token统计` 占位页
8. 平移 `说明文档`

这个顺序遵守了“先壳层、后挂载、最后接新增入口”的原则，能最大限度减少对成熟业务流的干扰。

## 12. 第一阶段风险点

### 12.1 测试集工作台结构迁移风险

当前测试集工作台内部本来围绕旧模式工作，第一阶段如果直接拆得太深，会打断已有联动关系。

### 12.2 `Prompt快速迭代` 深改前被提前耦合

它后续会大改，因此第一阶段如果不隔离入口，后面会再次牵动整个测试集工作台。

### 12.3 外部 `Chunking测试` 接入边界不清

如果直接把代码散拷进来，会给后续维护带来高成本。

### 12.4 用户路径变化

旧顶层入口消失后，必须保证用户在新结构里仍然能完整找到所有现有能力。

## 13. 第一阶段验证策略

第一阶段验证重点不是重新验证所有算法正确性，而是确认：

**平台结构变了，但能力没有丢。**

至少需要验证：

1. 一级导航 4 个入口都可进入
2. `测试集工作台` 3 个子页面都可切换
3. `模型结果验收` 的文件恢复、分析、导出仍可用
4. `Prompt自动优化` 仍可进入并执行原能力
5. `线上验证工作台` 仍可执行原摘录工作台流程
6. `数据预处理工作台` 中 `PDF压缩` 仍可用
7. `Chunking测试` 入口可进入真实页面
8. `Token统计` 占位页可见且定位清楚
9. `说明文档` 页面可正常显示

同时保持基础工程验证：

- `npm test`
- `npm run build`

## 14. 第一阶段明确不做的内容

第一阶段不做以下工作：

- 完整流程模式内部逻辑重构
- 全量去 ESG 化
- service / utils 全面重命名
- 全局状态管理改造
- 路由系统引入
- 真正可用的 `Token统计`
- 平台多租户、权限、协作能力
- 通用任务模板系统

这些内容属于后续阶段，而不是第一阶段壳层重组的一部分。

## 15. 完成标准

第一阶段完成时，应满足：

1. 产品顶部品牌已切换为 `LLM Lab`
2. 用户心智已从“垂直工具”切换为“实验平台”
3. 三大工作台与说明文档结构已成立
4. 现有成熟能力已在新结构中完成挂载
5. `Prompt快速迭代` 已被单独隔离为后续深改的入口
6. 第二阶段可以在不推翻第一阶段结构的前提下继续开发

## 16. 结论

第一阶段的本质不是功能重写，而是平台壳层重组。它要在尽量不扰动成熟业务链路的前提下，将项目从“ESG 报告摘录工具”安全演进为“以 ESG 为当前样例的大语言模型工程实验平台”，为后续的通用化和深改阶段建立稳定边界。
