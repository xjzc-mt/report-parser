## 项目概述

### 项目定位
**IntelliExtract** 是一个纯前端的 AI 驱动 ESG 报告数据提取工具，完全运行在浏览器中，无需后端服务器。

### 核心价值
- 从 PDF 格式的 ESG 报告中自动提取 700+ 个指标数据
- 支持多种 LLM（Gemini、OpenAI 兼容端点）
- 提供测试工作台进行双 LLM 对比和 Prompt 优化
- 内置 PDF 压缩功能

### 技术栈
```json
{
  "框架": "React 19 + Vite 5",
  "UI库": "Mantine 8",
  "PDF处理": "pdf.js 3.11.174 (CDN) + pdf-lib",
  "Excel处理": "xlsx",
  "压缩": "pako",
  "存储": "idb (IndexedDB)",
  "LLM SDK": "@anthropic-ai/sdk"
}
```

## 四大功能模块详解

### 1. 提取器（Extractor）

#### 功能描述
上传 PDF 报告 + Excel 需求文件 → AI 提取指标 → 导出结果到 Excel

#### 使用流程
1. 上传 PDF 文件（支持多个）
2. 上传需求 Excel（必须包含指标定义）
3. 配置 LLM 设置（模型、API Key、并发数等）
4. 选择指标类型（文字型/数值型/货币型/强度型）
5. 点击"开始提取"
6. 查看实时进度和日志
7. 导出结果到 Excel

#### 需求文件格式
Excel 必须包含以下列：
- `indicator_code`: 指标编码（唯一标识）
- `indicator_name`: 指标名称
- `value_type` 或 `value_type_1`: 指标类型
- `definition`: 指标定义（可选）
- `guidance`: 提取指导（可选）
- `prompt`: 提取提示词（可选，优先级最高）

#### 输出格式
导出的 Excel 包含：
- 所有输入列
- `text_value`: 提取的文本值
- `numeric_value`: 提取的数值
- `unit`: 单位
- `pdf_numbers`: 来源页码
- `found`: 是否找到（true/false）
- 汇总统计表（找到率、成本等）

---

### 2. 压缩器（Compressor）

#### 功能描述
纯浏览器端 PDF 压缩工具，基于 `pdf-lib` 实现

#### 实现原理
- 使用 `pdf-lib` 解析 PDF
- 移除嵌入字体、优化图片
- 使用 `pako` 进行 gzip 压缩
- 完全在客户端完成，无需上传服务器

#### 关键文件
- `src/services/pdfCompressorService.js`
- `src/components/CompressorTab.jsx`

---

### 3. 测试工作台（TestWorkbench）⭐ 核心功能

#### 功能描述
双 LLM 协作测试系统：
- **LLM 1**: 执行指标提取
- **LLM 2**: 分析错误并优化 Prompt

#### 完整工作流程

```
阶段一：提取与关联
├─ 1. 上传 PDF 文件列表 + 测试集 Excel（含标准答案）
├─ 2. 按 report_name + pdf_numbers 分组
├─ 3. 使用 pdfPageExtractor 提取指定页面为子 PDF
├─ 4. LLM 1 提取指标（固定 5 秒重试间隔，最多 3 次）
├─ 5. 右关联测试集 + 计算相似度
└─ 6. 自动下载"关联对比文件.xlsx"

[用户手动审核后点击"继续"]

阶段二：Prompt 优化
├─ 7. 按 indicator_code 跨报告分组
├─ 8. 为每份报告提取页面文本上下文
├─ 9. LLM 2 分析错误类型并生成改进 Prompt
└─ 10. 导出"最终优化结果.xlsx"
```

#### 测试集 Excel 必须包含的列

| 列名 | 说明 | 示例 |
|------|------|------|
| `report_name` | PDF 文件名（不含扩展名） | `2023年度报告` |
| `pdf_numbers` | 指标所在页码 | `12` 或 `12,13` |
| `indicator_code` | 指标唯一编码 | `E1.1` |
| `indicator_name` | 指标名称 | `温室气体排放总量` |
| `data_year` | 数据年份 | `2023` |
| `text_value` | **标准答案**（用于相似度对比） | `1234.56吨` |
| `value_type_1` 或 `value_type` | 指标类型 | `数值型` |
| `prompt` | 提取提示词（可选） | `提取范围1排放量` |

#### 指标定义文件（可选上传）

| 列名 | 优先级 | 说明 |
|------|--------|------|
| `indicator_code` | 必填 | 必须与测试集一致 |
| `prompt` | **最高** | 若存在，直接作为提取指令 |
| `definition` | 次要 | 仅在 `prompt` 缺失时使用 |
| `guidance` | 次要 | 仅在 `prompt` 缺失时使用 |

**优先级规则**：
1. 定义文件的 `prompt` 列（最高）
2. 测试集的 `prompt` 列
3. 定义文件的 `definition` + `guidance` 组合
4. 若以上都没有 → 报错

#### 错误类型分类（LLM 2 诊断）

| 类型 | 描述 | 优化策略 |
|------|------|----------|
| TYPE_A | 未找到（标准答案存在但 LLM 返回"未披露"） | 扩大搜索范围、补充同义词 |
| TYPE_B | 值错误（找到但内容不对） | 调整定位规则、增加优先级 |
| TYPE_C | 单位错误（主值对但单位错） | 明确"摘录原文单位" |
| TYPE_D | 过摘录（标准答案为空但 LLM 给出值） | 加强"未披露则返回未披露"约束 |
| TYPE_E | 年份错误（数值对但年份错） | 明确年份定位逻辑 |
| TYPE_F | 提取正确（相似度≥70%） | 保持或小幅润色 |

#### 相似度计算逻辑

```javascript
// 基于关键词匹配的相似度（0-100）
calculateSimilarity(testValue, llmValue)
  ├─ 完全相同 → 100
  ├─ 包含关系 → 100
  ├─ 分词 + 繁简转换
  └─ 交集/测试集大小 × 100
```

#### 支持恢复运行

任意阶段失败后：
- 抛出携带 `resumeState` 的 Error
- UI 层存储状态到 `localStorage`
- 用户点击"继续运行"时传回服务函数
- 从断点处继续执行

#### 独立优化入口

用户可直接上传"关联对比文件.xlsx"（跳过阶段一），直接运行 LLM 2 优化。

## 关键技术实现

### 1. 四种指标类型与批次拆分

#### 指标类型定义
```javascript
文字型 (text)    - 纯文本描述
数值型 (numeric)  - 数字 + 单位
货币型 (currency) - 金额 + 货币单位
强度型 (intensity) - 比率/密度（分子/分母）
```

#### 为什么要按类型拆分批次？
- 每种类型需要不同的系统提示词（System Prompt）
- 不同类型的提取逻辑和验证规则不同
- 提高提取准确率

#### 实现位置
- `src/utils/extraction.js` → `splitRequirementsIntoBatches()`
- `src/constants/extraction.js` → 四种类型的系统提示词

---

### 2. Worker 池并发控制

#### 为什么需要 Worker 池？
- 避免同时发起过多 API 请求导致限流
- 控制成本和速度平衡
- 提供可配置的并发数（默认 5）

#### 实现原理
```javascript
// src/services/extractionService.js
async function processExtractionBatch() {
  // 1. 按指标类型拆分批次
  const batches = splitRequirementsIntoBatches(requirements)
  
  // 2. 创建 Worker 池
  const workers = Array(parallelCount).fill(null)
  
  // 3. 并发处理批次
  await Promise.all(workers.map(async () => {
    while (批次队列未空) {
      const batch = 取出一个批次
      await processExtractionBatch(batch)
    }
  }))
}
```

### 3. Gemini vs 其他 LLM 的处理差异

#### Gemini 特殊处理
```javascript
// Gemini 接收原始 PDF 二进制（原生多模态）
if (isGemini) {
  const pdfBase64 = await fileToBase64(pdfFile)
  // 直接传 base64 给 Gemini API
}
```

#### 其他 LLM（OpenAI 兼容）
```javascript
// 其他 LLM 接收提取后的文本
else {
  const docText = await parsePDF(pdfFile)
  // 传文本给 API
}
```

#### 实现位置
- `src/services/llmClient.js` → `callLLMWithRetry()`
- `src/services/fileParsers.js` → `parsePDF()`, `fileToBase64()`

### 5. PDF 页面提取（测试工作台专用）

#### 功能
根据 `pdf_numbers` 字段（如 `"12,13"`）提取指定页面为独立 PDF

#### 实现流程
```javascript
// src/services/pdfPageExtractor.js
parsePdfNumbers("12,13") 
  → [12, 13]
  
extractPdfPages(pdfBytes, [12, 13])
  → 使用 pdf-lib 提取页面
  → 返回新 PDF 的 Uint8Array
  
uint8ArrayToBase64(pdfBytes)
  → 转为 base64 供 LLM 使用
```

#### 为什么需要？
- 减少 LLM 输入 token 数量（只发送相关页面）
- 提高提取准确率（减少干扰信息）
- 降低成本

## 数据流详解

### 提取器完整数据流

```
1. 用户上传
   ├─ PDF 文件 → pdfFiles[]
   └─ Excel 文件 → requirementsFile

2. 点击"开始提取"
   └─ App.jsx → handleStartExtraction()

3. 文件解析
   ├─ parsePDF(pdfFile) → docText (带 [Page X] 标记)
   ├─ parseExcel(requirementsFile) → requirements[]
   └─ fileToBase64(pdfFile) → pdfBase64 (仅 Gemini)

4. 数据规范化
   └─ normalizeRequirementRow() → 统一字段格式

5. 批次拆分
   └─ splitRequirementsIntoBatches()
       ├─ 文字型批次
       ├─ 数值型批次
       ├─ 货币型批次
       └─ 强度型批次

6. Worker 池并发处理
   └─ runExtractionJob()
       ├─ 创建 parallelCount 个 Worker
       └─ 每个 Worker 循环处理批次队列

7. 单批次处理
   └─ processExtractionBatch()
       ├─ 构建系统提示词（根据类型）
       ├─ 构建用户提示词（需求列表）
       ├─ callLLMWithRetry() → JSON 结果
       ├─ 解析 JSON
       ├─ mapBatchResults() → 映射回需求行
       └─ 返回 { batchResults, usage, batchCost }

8. 结果聚合
   └─ 合并所有批次结果 → results[]

9. 导出
   └─ exportResultsToExcel(results, stats)
```

### 测试工作台完整数据流

```
阶段一：提取与关联

1. 用户上传
   ├─ PDF 文件列表 → pdfFiles[]
   ├─ 测试集 Excel → testSetFile
   └─ 指标定义 Excel → definitionFile (可选)

2. 点击"开始运行"
   └─ TestWorkbenchTab.jsx → handleRunFullFlow()

3. 文件解析
   ├─ parseExcel(testSetFile) → testSetRows[]
   └─ parseExcel(definitionFile) → definitionRows[] (可选)

4. 合并定义（优先级处理）
   └─ mergeDefinitions()
       ├─ 定义文件的 prompt 列（最高优先级）
       ├─ 测试集的 prompt 列
       ├─ 定义文件的 definition + guidance
       └─ 若都没有 → 报错

5. 按报告分组
   └─ groupByReport(testSetRows)
       └─ { "报告A": [...], "报告B": [...] }

6. 逐报告处理
   └─ for each report:
       ├─ 查找对应 PDF 文件
       ├─ parsePdfNumbers(pdf_numbers) → [12, 13]
       ├─ extractPdfPages(pdfBytes, [12, 13]) → 子 PDF
       ├─ uint8ArrayToBase64(subPdf) → base64
       └─ 按指标类型分批

7. LLM 1 提取
   └─ callLLMWithRetry()
       ├─ 固定 5 秒重试间隔
       ├─ 最多 3 次重试
       └─ 返回 JSON 结果

8. 右关联测试集
   └─ joinTestSetWithLlm1()
       ├─ 以测试集为基准（右关联）
       ├─ 按 indicator_code 匹配
       └─ 未匹配的标记为"未提取"

9. 计算相似度
   └─ for each row:
       ├─ calculateSimilarity(test.text_value, llm.text_value)
       ├─ calculateSimilarity(test.numeric_value, llm.numeric_value)
       ├─ calculateSimilarity(test.unit, llm.unit)
       └─ similarity = 三者平均值

10. 导出关联对比文件
    └─ exportComparisonRows(comparisonRows)
        └─ 自动下载"关联对比文件.xlsx"

[用户手动审核后点击"继续运行"]

阶段二：Prompt 优化

11. 按指标跨报告分组
    └─ groupByIndicator(comparisonRows)
        └─ { "E1.1": [报告A, 报告B, ...], ... }

12. 逐指标处理
    └─ for each indicator:
        ├─ 收集所有报告的提取结果
        ├─ 提取页面文本上下文
        └─ 构建优化请求

13. LLM 2 分析与优化
    └─ callLLMWithRetry()
        ├─ 系统提示词：PROMPT_OPTIMIZER_SYSTEM_PROMPT
        ├─ 输入：多份报告的提取结果 + 标准答案
        ├─ 输出：错误类型诊断 + 改进 Prompt
        └─ 返回 JSON

14. 解析优化结果
    └─ for each indicator:
        ├─ error_types: ["TYPE_A", "TYPE_B"]
        ├─ pattern_analysis: "跨报告规律分析"
        ├─ improved_prompt: "改进后的提示词"
        └─ improvement_reason: "改进原因"

15. 导出最终结果
    └─ exportFinalResults(finalRows)
        └─ 下载"最终优化结果.xlsx"
```

### 独立优化模式数据流

```
1. 用户上传
   ├─ PDF 文件列表
   └─ 关联对比文件.xlsx（阶段一的输出）

2. 解析关联对比文件
   └─ parseComparisonFile()
       └─ 验证必需列存在

3. 直接跳到阶段二
   └─ runOptimizationPhase()
       └─ 同上述步骤 11-15
```

## 常见问题与解决方案

### 1. PDF 解析失败

**问题**：`pdf.js` 加载失败或解析错误

**解决方案**：
- 检查 `index.html` 中 CDN 链接是否正常
- 确认 PDF 文件未损坏
- 尝试使用其他 PDF 文件测试

**相关代码**：
```javascript
// index.html
<script src="https://cdnjs.cloudflare.com/ajax/libs/pdf.js/3.11.174/pdf.min.js"></script>
```

### 2. API 调用失败

**问题**：LLM API 返回 401/403/429 错误

**解决方案**：
- 检查 API Key 是否正确
- 检查 API URL 是否正确
- 检查是否超出配额或限流
- 降低并发数（parallelCount）

**相关配置**：
```javascript
// src/constants/extraction.js
DEFAULT_SETTINGS = {
  apiUrl: 'https://generativelanguage.googleapis.com/v1beta',
  apiKey: '',
  modelName: 'gemini-2.5-pro',
  parallelCount: 5,  // 降低此值可减少限流
  maxRetries: 3
}
```

### 3. Excel 解析失败

**问题**：需求文件或测试集解析错误

**解决方案**：
- 确认文件格式为 `.xlsx` 或 `.xls`
- 检查必需列是否存在
- 检查列名是否完全匹配（区分大小写）
- 确认没有合并单元格

**必需列检查**：
```javascript
// 提取器需求文件
['indicator_code', 'indicator_name', 'value_type']

// 测试工作台测试集
['report_name', 'pdf_numbers', 'indicator_code', 
 'indicator_name', 'data_year', 'text_value', 'value_type_1']
```

### 4. 页码提取错误

**问题**：测试工作台提取了错误的页面

**原因**：
- `pdf_numbers` 字段格式错误
- 物理页码 vs 内容页码混淆

**解决方案**：
- 确保 `pdf_numbers` 格式为 `"12"` 或 `"12,13"` 或 `"12-15"`
- 使用物理页码（PDF 文件的实际页数），不是内容中的页码标记
- 检查 PDF 是否有封面页导致页码偏移

**相关代码**：
```javascript
// src/services/pdfPageExtractor.js
parsePdfNumbers("12,13,15-17")
  → [12, 13, 15, 16, 17]
```

### 5. 相似度计算不准确

**问题**：明明内容相同但相似度很低

**原因**：
- 繁简体混用
- 标点符号差异
- 空格差异

**解决方案**：
- 系统已内置繁简转换
- 已忽略标点符号
- 如需调整算法，修改 `calculateSimilarity()` 函数

**相关代码**：
```javascript
// src/services/testBenchService.js
calculateSimilarity(testValue, llmValue)
  ├─ 繁简转换
  ├─ 分词（中英文混合）
  └─ 交集/测试集大小 × 100
```

### 6. 内存溢出

**问题**：处理大量 PDF 时浏览器崩溃

**解决方案**：
- 减少并发数（parallelCount）
- 分批处理 PDF 文件
- 使用页面提取功能（测试工作台）减少输入大小
- 关闭其他浏览器标签页

### 7. 恢复运行失败

**问题**：测试工作台中断后无法恢复

**解决方案**：
- 检查 IndexedDB 是否被清空
- 确认浏览器支持 IndexedDB
- 尝试清除缓存后重新运行

**相关代码**：
```javascript
// src/services/persistenceService.js
saveRunState(state)    // 保存状态
getRunState()          // 读取状态
clearRunState()        // 清除状态
```

- 推荐使用 Chrome/Edge（最佳性能）
- Firefox 可用但性能稍差
- Safari 可能有兼容性问题
- 不支持 IE

### 4. 文件大小限制
- 单个 PDF 建议 < 50MB
- Excel 文件建议 < 10MB
- 浏览器内存限制约 2GB

  // 示例：使用编辑距离

  return Math.round((1 - distance / maxLen) * 100);
