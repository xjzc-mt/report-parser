# 快速验收模式手动关联字段设计

## 1. 背景

当前测试集工作台的快速验收模式仅支持固定关联规则：

- `LLM.report_name = 测试集.report_name`
- `LLM.indicator_code = 测试集.indicator_code`
- `LLM.year = 测试集.data_year`

这套规则写死在服务层，适合标准输入，但不适合外部 LLM 结果文件字段命名不一致的场景。用户希望在快速验收模式中手动指定两份文件的关联字段，并在配置清空时自动退回当前默认规则。

## 2. 目标

本次只扩展快速验收模式，支持：

1. 用户手动配置 LLM 结果文件与测试集文件的关联字段。
2. 关联字段可增减，删除按钮位于每行末尾。
3. 关联字段候选项可直接来自上传文件中的实际列名。
4. 当手动配置被完全清空时，分析时自动回退到当前默认规则。
5. 点击分析按钮时执行严格校验，配置错误时阻止分析并给出明确提示。
6. 关联字段配置在快速验收模式内持久化恢复。

## 3. 范围

### 3.1 在范围内

- 快速验收模式 UI 增加手动关联字段配置入口。
- 快速验收模式服务层支持动态关联键。
- 快速验收模式持久化保存并恢复关联字段配置。
- 分析前的严格校验。

### 3.2 不在范围内

- 完整流程模式的关联逻辑改造。
- 快速优化模式的字段关联配置。
- 导出文件结构调整。
- 通用全工作台字段映射中心。

## 4. 交互设计

### 4.1 入口形式

在快速验收模式的两个上传卡片之间增加一个弱存在感的连接式触发器，不使用明显的大按钮。视觉目标是“连接两侧文件关系”的提示，而不是新的主操作入口。

交互特征：

- 默认折叠。
- 位于两个上传区之间。
- 可点击展开或收起。
- 展开后显示“关联字段配置面板”。

### 4.2 展开后的配置面板

每一行表示一组字段关联：

- 左侧：LLM 结果文件字段
- 中间：等号或连接符
- 右侧：测试集文件字段
- 行尾：删除按钮

面板底部提供“新增一组关联字段”按钮。

示意：

```text
[LLM结果文件]   [细连接图标]   [测试集文件]

展开后：
LLM列名            测试集列名
report_name    =   report_name      [删除]
indicator_code =   indicator_code   [删除]
year           =   data_year        [删除]

[新增一组]
```

### 4.3 默认映射

默认映射固定为：

- `report_name = report_name`
- `indicator_code = indicator_code`
- `year = data_year`

默认映射的用途：

1. 首次进入快速验收模式时，作为初始展示内容。
2. 当持久化配置为空时，作为默认展示内容。
3. 当用户将全部手动映射删空后，在分析时作为回退规则。

注意：删空并不表示“空配置生效”，而是表示“恢复系统默认关联规则”。

## 5. 字段来源与状态行为

### 5.1 字段候选来源

字段下拉候选项来自上传文件的实际表头。

实现方式：

- 上传 LLM 结果文件后，读取解析结果第一条记录的 key 集合作为 LLM 字段候选。
- 上传测试集文件后，读取解析结果第一条记录的 key 集合作为测试集字段候选。

不做额外全表扫描，仅使用解析后的列名集合，避免引入不必要性能负担。

### 5.2 文件未上传时的行为

- 若两侧文件尚未上传，面板仍可展示当前映射配置。
- 但字段选择控件不可用，或仅显示当前值不可编辑。
- 待文件上传并拿到列名后，字段选择才真正可操作。

### 5.3 文件变更后的行为

- 更换任一文件后，重新读取候选字段。
- 已保存映射仍然保留。
- 若保留的字段已不存在，严格校验会在分析前拦截。

这样可以避免用户更换文件时配置被静默重置。

## 6. 严格校验规则

点击“开始关联分析”时，先校验关联配置，再执行分析。

以下情况必须阻止分析：

1. 任一映射行左侧字段未选择。
2. 任一映射行右侧字段未选择。
3. 选择的字段不在当前上传文件的实际列名中。
4. LLM 侧存在重复字段映射。
5. 测试集侧存在重复字段映射。
6. 映射配置合法，但实际关联结果为 0 条匹配。

提示原则：

- 错误信息要直接指出问题，不使用模糊提示。
- 若有多项错误，可按第一条错误优先提示，保持交互简单。

推荐错误文案方向：

- `第 2 组关联字段未选择完整，请检查 LLM 字段和测试集字段。`
- `LLM 字段 "report_id" 不存在于当前文件中。`
- `测试集字段 "report_name" 被重复使用，请调整关联配置。`
- `当前关联字段配置无有效匹配，请检查两侧字段是否一一对应。`

## 7. 服务层设计

### 7.1 核心思路

将快速验收模式原先固定的 join key：

- `report_name + indicator_code + year/data_year`

改为：

- 默认映射模式：沿用当前固定规则
- 手动映射模式：按用户配置动态生成 join key

### 7.2 数据结构

关联字段配置建议使用数组结构：

```js
[
  { llmField: 'report_name', testField: 'report_name' },
  { llmField: 'indicator_code', testField: 'indicator_code' },
  { llmField: 'year', testField: 'data_year' }
]
```

默认映射也使用同样的数据结构，便于统一处理。

### 7.3 动态关联规则

服务层提供统一能力：

1. 归一化用户配置。
2. 若配置为空数组，则返回默认映射。
3. 按映射为 LLM 行和测试集行生成 join key。
4. 使用 join key 进行匹配。
5. 保留当前 comparison row 结构，避免影响统一分析页和导出逻辑。

### 7.4 0 匹配校验

“配置合法但 0 匹配”不属于 UI 静态校验，必须在服务层基于真实数据判断。服务层需要返回足够信息，使 UI 可以区分：

- 配置非法
- 配置合法但无有效匹配
- 配置合法且有匹配结果

## 8. 持久化设计

快速验收模式新增一份关联字段配置持久化数据。

要求：

- 进入页面时自动恢复。
- 修改配置后自动保存。
- 更换模式再切回时仍可恢复。
- 不覆盖现有快速验收文件恢复与结果恢复逻辑。

建议持久化内容：

```js
{
  fieldMappings: [
    { llmField: 'report_name', testField: 'report_name' },
    { llmField: 'indicator_code', testField: 'indicator_code' },
    { llmField: 'year', testField: 'data_year' }
  ]
}
```

特殊约定：

- 持久化允许保存空数组。
- 运行分析时如果读到空数组，则回退默认映射。

这样可以准确保留“用户主动清空”的状态，同时满足“清空即恢复默认规则”的业务要求。

## 10. 验证方案

### 10.1 逻辑验证

至少覆盖以下场景：

1. 不配置手动映射，使用默认规则成功关联。
2. 使用自定义字段映射成功关联。
3. 关联字段全部删空后，分析时回退默认规则。
4. 存在未选择字段时，阻止分析。
5. 存在字段不存在时，阻止分析。
6. 存在重复字段映射时，阻止分析。
7. 配置合法但 0 匹配时，阻止分析。

### 10.2 手工回归

手工验证快速验收模式：

1. 上传 LLM 结果文件与测试集文件。
2. 使用默认映射执行分析。
3. 修改为手动映射并重新分析。
4. 删除全部映射，确认仍可按默认规则分析。
5. 故意构造错误映射，确认分析前被拦截并提示。
6. 刷新页面，确认配置可恢复。

### 10.3 构建验证

提交前至少执行：

```bash
npm run build
```

## 11. 风险与约束

### 11.1 列名依赖首行解析

当前字段候选依赖解析结果中的字段集合。如果上传文件为空表，无法获得有效字段候选。此时分析应由现有解析或缺字段校验报错，不额外扩展空表特殊流程。

### 11.2 更换文件后的旧映射失效

用户更换文件后，旧映射可能引用不存在字段。本设计选择保留配置并在分析时拦截，而不是自动清洗，避免静默修改用户配置。

### 11.3 仅改快速验收模式

本次不抽象为全局关联配置中心，以控制范围，避免影响完整流程和快速优化模式的稳定性。

## 12. 结论

本设计通过“弱存在感映射入口 + 文件真实字段选择 + 严格校验 + 持久化恢复”的组合，在不破坏现有统一分析与导出结构的前提下，为快速验收模式提供可控的手动关联能力，并保留当前默认规则作为清空后的安全回退路径。
